# Onboarding Chat History Instructions

## Цель
Внедрить полноценную историю чата (user/assistant) для LLM-агента, чтобы GPT-4 видел весь контекст диалога и отвечал максимально естественно.

---

## 1. Анализ текущей архитектуры
- Вся логика flow реализована через последовательные узлы (nodes) в `root_graph.py`.
- Промпт для LLM собирается в `generate_llm_prompt` (`prompting.py`), сейчас он не учитывает историю сообщений.
- Состояние сессии хранится в MongoDB, структура описана в `Instructions.md` и `AppFlow.md`.
- В каждом цикле общения с LLM вызывается только один раз, история не сохраняется.

---

## 2. План внедрения истории чата

### Шаг 1. Расширить структуру состояния сессии
- В MongoDB-документе сессии добавить новое поле:
  - `history: List[Dict[str, str]]` — массив сообщений, где каждый элемент:
    - `{ "role": "user"|"assistant", "content": "..." }`
- Инициализировать пустым списком при создании новой сессии.

### Шаг 2. Модифицировать обработку сообщений
- В `chat_with_session` (endpoints/chat.py):
  - После получения сообщения пользователя, добавить его в `history` с ролью `user`.
  - После получения ответа от LLM, добавить его в `history` с ролью `assistant`.
  - Сохранять обновлённую историю в MongoDB после каждого шага.

### Шаг 3. Изменить генерацию промпта для LLM
- В `generate_llm_prompt` (`prompting.py`):
  - Вместо статичных блоков (system, user, assistant) собирать промпт так:
    1. System Prompt (один раз, как сейчас)
    2. Для каждого сообщения в `history`:
       - Если `role == "user"`: добавить строку `User: ...`
       - Если `role == "assistant"`: добавить строку `Assistant: ...`
    3. В конце — текущее сообщение пользователя (если оно ещё не добавлено в history)
    4. В самом конце — JSON Instructions (как сейчас)
- Убедиться, что порядок сообщений сохраняется.

### Шаг 4. Обновить тесты
- В тестах (`test_prompting.py` и других):
  - Добавить примеры с историей сообщений.
  - Проверить, что промпт корректно собирается из истории.

### Шаг 5. Проверить лимиты токенов
- Для длинных историй добавить проверку длины промпта (опционально, для будущего).
- Если история слишком длинная — обрезать старые сообщения (например, оставить последние 10 пар user/assistant).

### Шаг 6. Документировать изменения
- Описать новую структуру состояния и формат истории в `Instructions.md` и/или `README.md`.
- Добавить примеры истории чата и финального промпта.

---

## 3. Пример структуры истории
```json
"history": [
  { "role": "user", "content": "Hi, I'm Ben" },
  { "role": "assistant", "content": "Hello Ben! How old are you?" },
  { "role": "user", "content": "I'm 25" }
]
```

## 4. Пример финального промпта для LLM
```
System: You are a helpful coach.
User: Hi, I'm Ben
Assistant: Hello Ben! How old are you?
User: I'm 25
IMPORTANT: Respond in JSON format with ... (инструкции)
```

---

## 5. Важные моменты
- История должна сохраняться и обновляться после каждого шага.
- Промпт для LLM всегда строится на основе всей истории.
- JSON Instructions всегда идут в конце промпта.
- Для коротких flow (онбординг) лимит токенов не будет проблемой, но предусмотреть защиту на будущее.
